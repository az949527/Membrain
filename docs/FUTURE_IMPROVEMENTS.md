## RAG 管线深度优化

| 优化点 | 优先级 | 说明 | 建议时机 |
|--------|--------|------|----------|
| Reranker 精排 | 🟡 中 | 当前 FAISS 检索 top-k 后直接送入 LLM，缺少重排序环节。加一个 cross-encoder reranker（如 `BAAI/bge-reranker-v2-m3`）对检索结果精排，把最相关的排在前面，提高 LLM 回答质量 | 面试前 |
| Query 改写（HyDE） | 🟡 中 | 用户问题直接拿去向量检索效果有限，特别是短查询。HyDE 先用 LLM 根据问题生成一段"假设回答"，把假设回答向量化后再检索，能显著提升检索命中率。另一种思路是多轮改写：结合历史对话把"那他的公司呢？"改写为"张三所在的公司是什么？" | 面试前 |
| 多轮检索历史融合 | 🟡 中 | 当前每次检索只基于最新一条用户问题，不考虑对话上下文。更好的做法是：维护对话历史中的关键实体和主题，作为检索的辅助信号，避免每轮都"从零开始"检索 | 面试前 |

## Prompt Engineering 体系化

说明：当前 prompt 散落在各文件中（classify、entity_extractor、graph_retriever），未来统一管理时一并优化 prompt 设计体系。

| 优化点 | 优先级 | 说明 | 建议时机 |
|--------|--------|------|----------|
| System Prompt 结构化分层 | 🟡 中 | 将 system prompt 拆为 3 层：①角色定义（你是谁、什么风格）②能力边界（你有什么工具、能做什么）③行为约束（什么不能做、输出格式）。当前混在一起写，改一层动全部，分层后可独立迭代 | 面试前 |
| Few-shot 示例管理 | 🟡 中 | classify 节点和 Text2Cypher 的 few-shot 示例目前写死在 prompt 里。应抽离为独立模块，支持按场景加载不同示例集，并记录哪些示例效果好/差来持续优化 | 面试前 |
| CoT 推理引导 | 🟢 低 | 在 classify 节点中加入"先分析问题类型 → 再推理需要哪些知识源 → 最后输出决策"的 CoT 步骤，让 LLM 的决策更可解释，也便于调试。缺点是增加 token 消耗和延迟 | 面试前 |

## Embedding 选型与评估

说明：当前使用的 `shibing624/text2vec-base-chinese` 是项目初期"先跑起来"的选择，未经过系统性评估和对比。

| 优化点 | 优先级 | 说明 | 建议时机 |
|--------|--------|------|----------|
| 多模型对比评估 | 🟡 中 | 在固定测试集上对比 text2vec-base-chinese / BAAI/bge-large-zh-v1.5 / moka-ai/m3e-base 三个模型的检索准确率和召回率。评估指标：Recall@k、MRR、命中率。选最优的作为生产模型 | 面试前 |
| 领域微调方案 | 🟢 低 | 通用 embedding 模型在特定领域（如医疗、法律）表现偏差。如果项目有垂直领域需求，可用领域数据对 embedding 模型做继续预训练或对比学习微调，显著提升检索效果 | 后续 |
| 向量维度与性能平衡 | 🟢 低 | 当前使用 768 维向量。更高维度（如 1024）精度更好但检索更慢且存储更大。评估不同维度下的精度/性能/存储权衡，找到项目的最优平衡点 | 后续 |
